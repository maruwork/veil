# トークン最適化ポリシー（一般ルール）

**目的**: エージェントのトークン無駄を減らす。ただし、正本の読みやすさ・権限境界・リポジトリ構造を弱めない範囲で、今の作業を助ける制御だけを採る。

この policy は
`../frameworks/project-progression-rule.md`
のうち、主に

- `正本を補助面で置き換えない`
- `入口を薄く保つ`
- `source-first で読む`

を、トークン最適化の面で補助する文書である。

> **正本の関係**: この file は `pj-template` の共通 policy 棚にある正本である。project ごとの token 最適化運用は、この文書を基準に local rule へ具体化する。

## 1. 採用している制御

そのプロジェクトが今、プロジェクト全体の作業ルールとして採るトークン最適化はこれだけ。

1. 繰り返すシェル出力の圧縮
2. ルート入口ファイルを薄く保つ
3. 境界優先の検索・読み込み（まず入口・境界から読む）
4. compact / clear / plan を意識的に使い分ける

目的は「最大限の圧縮」ではなく、「正本の信号を失わずにトークンを減らす」こと。

## 2. 定型出力圧縮

- シェル出力を圧縮するための repo 外補助ツールや wrapper は使ってよい。
- その補助ツールを正本（authority）の面にはしない。
- その補助ツールは次を置き換えない：現行タスク正本 / ワークフロー・タスク・設計の正本 / DB・実行時の真実。

運用の詳細は、各 project 側で定義された runtime token optimization note or manual に落とす。

## 3. 入口ファイルを薄く保つ

- entry file は入口であって、長いマニュアルにしない。
  - 代表例を出す場合も 1 箇所の注記に留め、template の半既定に見せない。
- 安定したルールは、template 共通 policy 棚または対象 project 側の governance / docs 棚に置く。
- 繰り返す手順は、ルート入口を膨らませず、再利用できる policy / manual / checklist / spec の棚へ移す。

## 4. 境界優先の検索・読み込み

- まず出どころ（source）と棚から読む。
- 広い再読み込みの前に、次を優先する：entry file / current surface / 境界レジスタ / 自動生成の投影。
- 全体一括の取り込みより、的を絞った検索と狭い範囲読みを優先する。
- トークン最適化のためだけに、新しいルート直下の ignore / 制御ファイルを増やさない（別途承認がある場合を除く）。
- 既存のリポジトリ境界と配置ルールが、その project 内のトークン制御の主役。

## 5. compact / clear / plan の使い分け

- 「1つの区切られたタスク／1つの区切られたレビュー」を既定の作業単位にする。
- 広い複数ファイル作業の前に、まず計画する。
- compact するのは：目的が変わったとき / 同じ説明を繰り返し始めたとき / 同じ正本の文脈を何度も払い直しているとき。
- clear / 再起動するのは：今のセッションが区切られた作業を反映しなくなったとき。

## 6. まだ採らないもの（保留）

次は、今はその project 全体の必須ルールには採らない。

- 広域 context 圧縮ツール / 外部知識補助 / proxy 層 / budget 制御層 / モデルルーティングをリポジトリ docs に持つこと

理由：リポジトリ外の実行時状態に依存する / 今の利点に対して運用面が大きすぎる / 区切った将来の採用作業として扱う方がよい、のいずれか。

## 7. 境界ルール

- トークン最適化は、第二の正本面を作ってはいけない。
- 圧縮ツールは量を減らしてよいが、次を再定義してはいけない：真実 / タスク状態 / 判定 / 持ち主 / 正本の配置。
- トークン節約の手法が正本の読みやすさと衝突するなら、正本の読みやすさを優先する。
