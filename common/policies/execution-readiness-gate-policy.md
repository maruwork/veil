# 着手前確認と実行ルール

**目的**: 本当に次へ進めてよいか、設計が足りているか、どこで止まるかを 1 本で確認するための共通基準。

この policy は
`../frameworks/project-progression-rule.md`
のうち、主に

- `進んでよい条件`
- `着手前確認`
- `停止条件`
- `checkpoint 前進前の確認`

を具体化する。

1つでも曖昧、未確認、未達のままなら進めない。

この policy では、各大項目について `何で確認したか` も必ず答えられなければならない。

- 何を見て確認したか
- どの file / rule / command / output を根拠にしたか
- その証拠をどこへ残すか

## 1. ゴール

次が明確か。

- 完成状態
- 今回の範囲
- 範囲外
- 未達条件
- この作業で終わること
- この作業で終わらないこと
- 完了報告の主語

## 2. 道のり

次が明確か。

- どの順で進むか
- どこで止まるか
- 前後の依存関係

## 3. チェックポイント

次が明確か。

- どのチェックポイントがあるか
- どの順で通るか
- 各チェックポイントが何のズレを防ぐか
- 通過前後の成立条件

## 4. タスク

次が明確か。

- 各チェックポイントに必要なタスクがそろっているか
- 抜け漏れがないか
- 別のチェックポイントの作業が混ざっていないか
- 順番が正しいか

## 5. 設計

各タスクについて次がそろっているか。

- 着手条件
- 読む場所
- 書く場所
- 触ってはいけない場所
- やること
- 出力
- 合格条件
- 失敗条件
- 停止条件
- 証拠
- 記録先

各タスクの設計には、少なくとも次を入れる。

1. `Task ID`
2. `親テーマ`
3. `親チェックポイント`
4. `active bundle id`
5. `active bundle type`
6. `成功主語`
7. `今回やる範囲`
8. `今回やらない範囲`
9. `目的`
10. `このタスクが必要な理由`
11. `着手条件`
12. `入力`
13. `読んでよい場所`
14. `書いてよい場所`
15. `触ってはいけない場所`
16. `やること`
17. `期待する出力`
18. `合格条件`
19. `失敗条件`
20. `停止条件`
21. `差し戻し条件`
22. `人判断へ上げる条件`
23. `証拠`
24. `結果の記録先`
25. `最終判定者`

## 6. 実行面

次があるか。

- 実際に作業する場所
- 結果を残す場所
- 記録と証拠を残す場所

## 7. test / tool / command

次が明確か。

- 必要な test
- 必要な tool / script
- 使う command

## 8. 前提条件

次が明確か。

- 環境変数
- DB 接続
- サービス
- 権限
- fixture / data

足りないものは blocker として名前付きで書く。

## 9. 各層の連結

次が切れていないか。

- `ゴール -> 道のり`
- `道のり -> チェックポイント`
- `チェックポイント -> タスク`
- `タスク -> 設計`

## 10. 完了境界

次が言えるか。

- 全タスク完了で全チェックポイント通過になる
- 全チェックポイント通過で道のり完了になる
- 道のり完了でゴール達成になる
- 今回の完了と後続作業の境界が切れている
- 完了報告の主語が途中で変わらない

## 11. 担当と範囲

次が明確か。

- owner
- 今回の範囲
- 今回やらない範囲
- 実行順

## 12. 失敗時の扱い

次が明確か。

- どの条件で止まるか
- どの条件で blocked にするか
- どの条件で差し戻すか
- どの条件で再開できるか
- 失敗時に何を残すか

次の前には必ず止まる。

- 正本ファイルの移動、削除、rename、archive
- root entry、rule、taxonomy、boundary、disposition の意味変更
- DB write や永続的な外部 system への write
- credential、secret、有料 API、private remote service の使用
- branch / merge / release / deployment policy の変更
- 所有境界の変更
- temporary output や外部入力を正本へ昇格する時
- 互換しない将来運用モデルの選択

次は止まらず進めてよい。

- read-only の調査
- local lint / test / consistency check
- 既存ルールを再定義しない追記
- reversible な branch-local 修正
- report、分類表、実行計画

止まる時は最低限、次を書く。

- 停止理由
- 必要な判断
- 選択肢
- 推奨
- 判断材料
- 再開条件
- 停止中でも進めてよい作業があるか
- あるならその範囲
- 次に必ず触るファイル
- このターンで最低限埋める範囲
- 判断後の次アクション

## 13. 結果の確認と反映

次が明確か。

- 何を見て結果を確認するか
- どこに結果を記録するか
- どこに今の状態を反映するか
- current と historical をどう分けるか

## 14. 後ろまで見えているか

次が明確か。

- 今回どこまで進めるか
- 次に何をやるか
- 後続に必要な前提が揃っているか
- 後ろで止まる見切り発車になっていないか

## 15. 必要な承認と巻き戻し手段

次があるか。

- 必要な承認
- 必要な権限
- 失敗時の巻き戻し手段

## 16. 合格条件

次をすべて満たす時だけ進めてよい。

- `ゴール、道のり、チェックポイント、タスク、設計` がつながっている
- `active bundle id` と `active bundle type` が固定されている
- `成功主語`、`今回やる範囲`、`今回やらない範囲` が固定されている
- 完了報告の主語が固定されている
- 実行面と結果保存面がある
- 必要な test / tool / command がある
- 必要な前提条件が確認済み、または blocker として明記されている
- 担当、範囲、順番、失敗時の扱いが明確
- 各大項目について evidence source が答えられる
- `close 済み bundle` と `post-close residual` を混ぜていない
