# ファイル操作ポリシー（一般ルール）

**目的**: ファイルの作成・移動・保管（archive）・修正を行うための、どのプロジェクトでも使える一般ルール。プロジェクト固有の置き場所や保護対象は、各プロジェクトが定義する。

> **正本の関係**: この file は `pj-template` の共通 policy 棚にある正本である。project 固有の置き場所や保護対象は、この一般ルールの下で local rule として追加する。

この policy は `作成 / 移動 / archive / 処分` の操作境界を扱う。

- 棚名や file 名の付け方そのものは
  `naming-and-shelf-policy.md`
- 入口 / guide / reference の分離は
  `entry-guide-reference-separation-policy.md`

を正本として参照する。

## 1. まず置き場所を決める

- ファイルを作る前に棚を決める。
- 新しいフォルダより、既存の棚を優先する。
- 既存の棚で役割を説明できないときだけ、新しいフォルダを作る。
- 新しいフォルダは、1ファイルだけでなく複数ファイルや繰り返す作業に役立つこと。

## 2. フォルダを作ってよい条件

- 役割が既存フォルダと明確に違う。
- 入る中身に一貫性があり、継続して発生する。
- 配置表や案内で説明できる。
- 何でも放り込む未整理の置き場にならない。

## 3. 移動・保管のルール

- 移動する前に、参照元・呼び出し元を確認する。
- 履歴価値があるものは、削除より保管（archive）を優先する。
- 保管したファイルが現役から外れた理由を記録する。
- 現役ファイルと、保管・一回限りのファイルを同じ棚に混ぜない。

## 4. 一時作業

- 一時出力には、宣言済みの作業用フォルダ（workspace / scratch）を使う。
- 一時出力を正式ドキュメントとして扱わない。
- 一時作業を昇格するときは、レビューと配置判断を通す。
- 作業用フォルダが複数あるときは、新しい一時出力は1つの現役 workspace に寄せる。

## 5. 変更後の追随

ファイルの役割や場所が変わったら、いちばん近い次のものを更新する。

- 配置表や配置マップ
- 案内（index）
- 境界・処分の記録
- 呼び出し元・設定の参照

追随先のドキュメントが無いときは、記憶に頼らず小さな記録を作る。

## 6. 処分の考え方

原則は **archive を優先し、削除は最後**。

ファイルを動かす・消す前に、次の観点で見る。

- 今の実行で使われているか
- 実際に参照されているか
- 今後も保守する意思があるか
- 誤読・変更したときの影響は大きいか
- 履歴・再検証としての価値があるか

区分は次の 5 つだけに絞る。

- `current keep`
- `exception keep`
- `isolated keep`
- `archive keep`
- `delete candidate`

未分類のまま現役の棚に残さない。archive を今の状態として扱わない。

## 7. 命名・パス・保護

- 現役・正本の file には安定した名前を使う
- 現役・正本の file 名に日時を入れない
- log・snapshot・書き出し・report・保管物には日時を付けてよい
- 再利用する code に利用者ごとの絶対 path を直書きしない
- test や一時出力には temporary directory を使う
- project 外への書き込みは明示承認なしで行わない

次のような高リスク file は保護対象として扱う。

- governance / policy
- schema / migration 履歴
- append-only audit record
- production config
- root entry の指示書

秘密情報を追跡 file や log に入れない。大きい生成物・binary は必要な場合を除き source 管理に入れない。
