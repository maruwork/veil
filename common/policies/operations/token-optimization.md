# Token Optimization Policy

**Purpose**: Reduce token waste in agent work without weakening canonical readability, authority boundaries, or repository structure.

This policy supports the token-optimization side of `../../frameworks/core/progression-rule.md`, especially:
`do not replace canonical sources with helper surfaces`, `keep entry surfaces thin`, and `read source-first`.

> **Canonical relationship**: This file is the canonical shared policy in `pj-template`. Each project may add local token-optimization rules under this baseline.

## 1. Controls We Adopt

These are the only shared controls adopted here:

1. compress repetitive shell output
2. keep root entry files thin
3. search and read from boundaries first
4. use compact / clear / plan deliberately

The goal is to reduce tokens without losing canonical signal.

## 2. Default Adoption Rule

When a project adopts token optimization under `pj-template`, do not ask for the "best" file from `common/token-optimization` on each run.
Use this fixed default order instead and install only the smallest required local set:

1. create or update the project-local runtime token note from `AI_AGENT_RUNTIME_TOKEN_OPTIMIZATION.md`
2. if root `AGENTS.md` is absent, install it from `AGENTS.md.token-optimized`
3. if root `CLAUDE.md` is absent, install it from `CLAUDE.md.token-optimized`
4. add `MEMORY` or continuation support only when the project explicitly adopts them
5. do not copy the shared `README.md` or `manuals/` into project-local authority unless the owner explicitly asks for a localized copy

This keeps adoption deterministic and prevents repeated case-by-case selection work.

## 3. Template Output Compression

- External helper tools or wrappers that compress shell output are allowed.
- Do not treat them as canonical surfaces.
- They must not replace the canonical source for the current task, workflow, tasking, design, database, or runtime truth.

Operational details belong in project-local runtime notes or manuals.

## 4. Keep Entry Files Thin

- An entry file is an entry surface, not a long manual.
- If examples are needed, keep them to one representative note or pointer.
- Stable rules belong in shared template policies or the target project's governance or docs shelves.
- Repeated procedures should move into reusable policies, manuals, checklists, or specs instead of inflating the root entry file.

## 5. Search and Read from Boundaries First

- Start from the source and its shelf.
- Before broad rereads, prefer entry files, current surfaces, boundary registers, and autogenerated projections.
- Prefer targeted search and narrow reads over bulk ingestion.
- Do not add new root-level ignore or control files only for token optimization unless separately approved.
- Existing repository boundaries and placement rules should remain the primary token-control mechanism inside the project.

## 6. Use Compact / Clear / Plan Deliberately

- Treat one bounded task or one bounded review as the default work unit.
- Plan before large multi-file work.
- Use compact when the objective changes, explanations start repeating, or the same canonical context keeps getting repaid.
- Clear or restart only when the current session no longer reflects the bounded work in progress.

## 7. Items Not Adopted Yet

The following are intentionally not mandatory project-wide rules at this stage:

- broad context-compression tools
- external knowledge helper layers
- proxy layers
- budget-control layers
- model-routing logic in repository docs

They remain out of scope because they would depend too much on external runtime state, add operating weight, or deserve a separate adoption track later.

## 8. Boundary Rule

- Token optimization must not create a second canonical surface.
- Compression tools may reduce volume, but they must not redefine truth, task state, judgments, owners, or canonical placement.
- If a token-saving tactic conflicts with canonical readability, canonical readability wins.
