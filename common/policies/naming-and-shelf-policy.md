# 命名と棚ポリシー（一般ルール）

**目的**: プロジェクト固有の配置表を当てる前に使う、移植できる命名・棚の一般ルール。

> **正本の関係**: この file は `pj-template` の共通 policy 棚にある正本である。project 固有の命名例外や棚配置例外は、この一般ルールの下で local rule として追加する。

この policy は `命名` と `棚を増やしてよい条件` を扱う。

- 実際に file をどう動かすか、archive するか、処分するかは
  `file-operation-policy.md`
- entry / guide / reference の読み面分離は
  `entry-guide-reference-separation-policy.md`

を正本として参照する。

## 命名

- 現役・正本のドキュメント名に日時を入れない。
- 生成物・ログ・スナップショット・書き出し・レポート・保管物には日時の接尾辞を付けてよい。
- 新しい移植可能なドキュメント名は小文字ケバブケース（lowercase-kebab-case）を推奨する。
- コード・設定のファイル名は、その言語やツールの慣習に従う。
- プロジェクト固有の名前は、共通の棚ではなくプロジェクトの棚に置く。
- 取り込んだ／過去のファイル名は、正式にそろえる作業をするまで元のままでよい。
- 既存の例外名を改名するときは、見た目より先に、呼び出し元・記録・マニュアル・生成物への影響を確認する。

## フォルダ命名

- 新しい共通の棚のフォルダ名は小文字ケバブケースを原則とする。
- 保管・取り込み・過去のフォルダ群は、復元の手がかりや元の名前を失わない範囲で元のままでよい。
- 番号付きの並びは `01-...` のようなゼロ詰め接頭辞を許すが、意味のない連番を新設しない。

## 入口ファイル

- 棚の入口は原則 `README.md` を使う。
- `INDEX.md` は、取り込み・過去・レガシー書き出しや、呼び出し元がその名前を前提にする場合だけ許す。

## 接頭辞・言語の混在

- `PT-` `SE-` `ADR-` のような接頭辞は、番号体系そのものが意味を持つ棚だけで使う。
- 新しい共通ドキュメントで、必要のない大文字 ID 接頭辞を付けない。
- 日本語と英語の混在ファイル名は、取り込み・過去・プロジェクト固有の案内では許すが、新しい共通の再利用資産では避ける。

## 改名の判断

- 改名は、見た目より、呼び出し元・記録・マニュアル・生成物・フック・テスト・DB への影響を優先する。
- 影響が小さいものだけ、すぐ改名してよい。
- 影響が大きいものは「要協議の改名候補」として記録し、相談前提で扱う。

## 棚のルール

- どのプロジェクトでも使える再利用資産は、共通の棚に置く。
- プロジェクト固有のガバナンス・強制ルールは、そのプロジェクトのガバナンス棚に置く。
- 外部からの調査入力は、正式に昇格するまで、プロジェクト内に重複した取り込み先を作らない。
- 新しい棚を作る前に、既存棚の README / guide / register がその役割を既に持っていないか確認する。
- 棚は「置き場所がほしい」ではなく、既存棚と異なる reader role / authority role がある時だけ増やす。
- 棚を増やす場合は、棚の入口、戻り先、current / generated / historical の扱いを 1 つの README か register に明示する。
- 同じ reader role の棚が 2 つ以上ある場合は、新設ではなく統合・rename・archive 候補として扱う。

## 保管と削除

- 削除より保管（archive）を優先する。
- 履歴価値はあるが現役の役割がないファイルは保管する。
- 削除は、生成物・ローカル状態・履歴価値のない重複・明示承認済みの場合だけ。

## 共通化

プロジェクト固有ファイルに再利用できる原則が含まれるときは、元ファイルを動かさず、移植版を共通の棚に抜き出す。別の移行作業で決まるまで、元ファイルはプロジェクト固有の正本として残す。

## 肥大化防止

- ファイルやフォルダを追加する時は、同時に削れる重複・古い説明・不要な route note がないか確認する。
- 新設より先に、既存の棚で `entry / guide / reference / canonical / evidence / archive` のどれに当たるかを分類する。
- 分類できないものは canonical shelf に置かず、temporary / workspace / draft の扱いにする。
- 長期的に残す棚は、reader が「ここに何があるか」「ここに何がないか」を README だけで判断できる状態に保つ。
