# VOCAB LAYER 設計書 v2

> **アーカイブ**: この設計書は v0.1〜v0.2 時点のスナップショットです。現在の実装とは乖離があります。最新の設計は `veil-design.md` を参照してください。

**プロダクト名：** VOCAB LAYER  
**種別：** AIから人間への出力適応エンジン（受け取り側語彙変換）  
**バージョン：** 0.2（MVP実装済み）  
**作成日：** 2026-05-27

---

## 1. 解決する問題

AIが生成したテキストには英語・技術用語・プロジェクト固有ラベルが混在する。受け取る人間がそれぞれの語を「説明語なのか、変数名なのか、固定値なのか」を判断するたびに認知コストが発生する。

本ツールはこの問題を「受け取り側」で解決する。モデルを変更せず、AIと人間の間に変換層を置く。

**本質：「一般的にわかりやすい」ではなく「この人にとってわかりやすい」を実現する。**

---

## 2. 設計方針

- **単語の置き換えのみ行う。AIによる文章の再構成は行わない。** 一貫性を保つため。
- **分類を先に行い、置き換え対象外の語はスキップする。** ファイル名・変数名・固定値を誤って変換しない。
- **使いながら育てる設計。** 最初から完璧な語彙DBを作らない。選択するたびに蓄積される。
- **個人に合わせる。** 汎用的なわかりやすさではなく、この人固有の語彙選好をDBに育てる。
- **用語辞書として機能させる。** 蓄積した語彙はプロジェクトをまたいで活用できる資産。

---

## 3. 語彙の分類体系

| 分類ID | 名称 | 説明 | 例 |
|---|---|---|---|
| 1 | 説明語 | カタカナ・日本語に言い換え可能 | active、close、blocker |
| 2 | 固定値 | status値・verdict値。そのまま表示 | "ACTIVE"、"continue" |
| 3 | 変数名・フィールド名 | コード上の固有名。そのまま表示 | `active_branch`、`parent_verdict` |
| 4 | ID | 定義済みパターン。そのまま表示 | BRANCH-046、PARENT-006 |
| 5 | 固有名詞・製品名 | 個人によって扱いが変わる | PostgreSQL、pytest |
| 6 | プロジェクト固有用語 | 一般語でもコード名でもない概念語 | canon |
| 7 | 境界が曖昧なもの | 文脈によって分類が変わる | reroute |

**置き換え対象：分類 1 / 5 / 6 / 7**  
**置き換え対象外：分類 2 / 3 / 4**

---

## 4. 変換候補の形式

語ごとに3パターンを保持する。

| フィールド | 説明 |
|---|---|
| p1 | 日本語1（主変換先） |
| p2 | カタカナ |
| p3 | 日本語2（別表現） |

登録時に空欄の候補はハイライトクリック時にClaude APIが自動生成して補完する。選択した候補はDBに自動保存される。

---

## 5. スキップ判定ロジック

以下のいずれかに該当する場合、語の置き換えをスキップする。

**正規表現レベル（lookahead / lookbehind）**

```
(?<![A-Za-z0-9_.-]) 語 (?![A-Za-z0-9_.-])
```

これにより以下を自動除外する。

- スネークケース内の語（`_`で前後に連結）例：`workflow_aggregate_packet.py`
- ファイル名内の語（`.`で拡張子に連結）
- ハイフン区切りのパス内の語（`-`で連結）例：`task-pier-workflow-implementation-wave.md`
- `key=value`形式の値部分

**isProtected関数レベル（コンテキスト判定）**

- バッククォートで囲まれた範囲内
- ダブルクォートで囲まれた範囲内
- `=`の直後

## 6. 変換処理フロー

```
入力テキスト（テキストエリアに貼り付け）
↓
「変換する」ボタン押下
↓
[1] 語彙DBから置き換え対象（分類1/5/6/7）を抽出
↓
[2] 長い語句から順にソート（複合語を先に処理）
↓
[3] 正規表現でマッチ → isProtected判定
↓
[4] 置き換え対象 → p1（日本語1）に置き換え
    スキップ対象 → そのまま
↓
変換前／変換後の2列表示（行単位で対応）
↓
ハイライト語をクリック
↓
[5] 登録済みp1/p2/p3を確認
    未登録の候補 → Claude APIで自動生成
↓
カタカナ／日本語1／日本語2／自分で設定の4候補を表示
↓
選択 → DBに保存 → 次回から即時表示
```

## 7. 語彙DBスキーマ

**現行：メモリ上（リロードで消える）**  
**移行先：SQLite（永続化）**

```sql
CREATE TABLE vocab (
  id         INTEGER PRIMARY KEY AUTOINCREMENT,
  original   TEXT UNIQUE NOT NULL,  -- 元の語（英語）
  p1         TEXT DEFAULT '',       -- 日本語1
  p2         TEXT DEFAULT '',       -- カタカナ
  p3         TEXT DEFAULT '',       -- 日本語2
  cat        INTEGER DEFAULT 1,     -- 分類ID（1-7）
  use_count  INTEGER DEFAULT 0,     -- 使用回数
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

## 9. 現時点の課題と対応方針

### 優先度：高

| 課題 | 状況 | 対応方針 |
|---|---|---|
| DBがメモリ上 | リロードで語彙が消える | SQLite化（最優先） |
| 登録済みリストに検索なし | 語彙が増えると使えない | 検索ボックス追加 |
| 「自分で設定」で選んだ語がDBに反映されているか不明 | 確認手段がない | 選択後にリスト更新して確認可能にする |

### 優先度：中

| 課題 | 状況 | 対応方針 |
|---|---|---|
| エクスポートなし | 別環境に持ち出せない | JSON/CSVエクスポート機能 |
| インポートなし | 他プロジェクトの辞書を取り込めない | JSONインポート機能 |
| プロジェクト別辞書なし | Pier用・AIM用を切り替えられない | プロジェクト切り替えUI |
| カタカナ生成精度 | 過去形・進行形を原形に変換できていない | プロンプト改善（継続） |

### 優先度：低

| 課題 | 状況 | 対応方針 |
|---|---|---|
| 使用頻度順・登録順の切り替えなし | ソート固定 | ソートUI追加 |
| カテゴリフィルタなし | 全語彙が混在 | カテゴリ別フィルタ |
| 変換後テキストのコピーボタンなし | 手動コピーが必要 | コピーボタン追加 |

## 10. Claude Codeへの引き継ぎ事項

### 現在の実装ファイル
- `vocab_converter.html` ── 単一HTMLファイル（メモリDB・フロントエンドのみ）

### 次フェーズの実装方針

**SQLite化**
- Pythonバックエンド（`app.py`）でSQLiteを管理
- フロントエンドからREST APIで語彙を取得・更新
- `/vocab`（GET/POST）、`/convert`（POST）、`/delete`（POST）

**検索機能**
- サイドバーの登録済みリスト上部に検索ボックスを追加
- クライアントサイドでリアルタイムフィルタリング（SQLite化後はサーバーサイド検索も可）

**エクスポート/インポート**
- JSON形式でvocab配列をそのまま出力・読み込み

### 注意事項
- スキップ判定ロジック（`isProtected`関数）は変更しないこと。ファイル名・変数名の誤変換を防ぐ核心ロジック。
- 長い語句を先に処理する順序（`sort((a,b) => b.o.length - a.o.length)`）は必ず維持すること。`stale blocker`より先に`stale`を処理すると複合語が壊れる。
- Claude APIへのリクエストはフロントエンドから直接呼び出す現行方式を維持してよい。
