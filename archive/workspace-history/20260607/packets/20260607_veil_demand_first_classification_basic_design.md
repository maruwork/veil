# VEIL 高需要先行判別ルール 基本設計

Status: Draft
Date: 2026-06-07

## 1. 目的

候補語の扱いを「一気に全部登録する」流れから、「判別して、需要が高いものから段階的に採用する」流れへ整理し、VEIL の運用面を安定させる。

## 2. 境界

- 正本:
  - `AGENTS.md`
  - `README.md`
  - `docs/veil-design.md`
  - `docs/manual.html`
  - `skills/codex/veil-capture/SKILL.md`
  - `skills/claude-code/veil-capture.md`
- 非対象:
  - runtime ロジック
  - UI
  - `common/`
  - `index/`

## 3. 採用する考え方

### 3.1 判別を先に行う

候補語は、訳語決定の前に次の順で切る。

1. 識別子
2. 固有名
3. 説明語
4. プロジェクト固有語
5. 境界が曖昧な語

これにより、変換してはいけないものと、変換すべきものを先に分ける。

### 3.2 高需要先行で採用する

採用対象は、次の順で押し込む。

1. 頻出で困り度が高い語
2. VEIL 運用の基幹語
3. プロジェクト固有語
4. 低頻度語と曖昧語

この順で採用すると、少数のルールで効きが出やすい。

### 3.3 保留を失敗ではなく待機として扱う

保留は未完了ではなく、次回判定へ送る待機棚として扱う。

保留条件:

- 固有名か一般語か迷う
- 候補1が競合する
- 既存ルールと別訳衝突する
- 頻度も困り度もまだ低い

## 4. 文書反映方針

### 4.1 README

- VEIL の運用原則として「高需要語から段階導入」を追加する
- 使い方に「全部を一度に登録しない」方針を短く書く

### 4.2 AGENTS

- delegated AI が最初に読む面として、高需要先行と保留運用を短く固定する

### 4.3 設計書

- 判別ルールをカテゴリ設計と接続する
- 採用優先度と保留条件を、capture の処理フローへ組み込む

### 4.4 UI マニュアル

- カテゴリ自動推定は入口であり、即採用ではないことを書く
- 高需要語から段階導入する順序を UI 利用者にも見えるようにする

### 4.5 capture 手順

- 候補抽出後に「判別」と「優先度付け」を明文化する
- 書き込み前に「今回採用する語」と「保留へ送る語」を分ける

## 5. 却下する案

- 全候補を毎回登録対象にする案
  - 正本が早く膨らみ、見直しコストが上がるため不採用
- 判別より先に訳語決定へ進む案
  - 識別子や固有名まで登録しやすくなるため不採用
- 需要判定をコード化して今回実装する案
  - 今回の範囲は運用ルールの固定であり、runtime 変更は外す
