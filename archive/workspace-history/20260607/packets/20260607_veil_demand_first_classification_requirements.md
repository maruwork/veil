# VEIL 高需要先行判別ルール 要件

Status: Draft
Date: 2026-06-07

## 1. 目的

VEIL の語彙運用で、候補語を一気に判別・登録するのではなく、需要が高いものから段階的に採用する方針を正本へ反映する。

今回の要件は、判別ルール、採用優先度、保留運用を README / 設計書 / capture 手順 / 入口文書 / UI マニュアルへ整合して書き下ろすことである。

## 2. 背景

- 需要が固まっていない語まで早く正本化すると、見直しコストが増える
- 抽出語の揺れを統合しても、採用優先度が曖昧だと正本が育ちにくい
- 識別子、固有名、説明語、プロジェクト固有語、境界が曖昧な語の扱いを先に決めないと、capture の判断がぶれやすい

## 3. 今回の範囲

- 語の判別順序を定義する
- 採用優先度を定義する
- 保留へ回す条件を定義する
- 既存の `veil-capture` 手順へ段階導入の順序を入れる
- README と設計書に、VEIL の運用原則として反映する
- `AGENTS.md` に delegated AI 向けの運用原則を反映する
- `docs/manual.html` に UI 利用者向けの段階導入説明を反映する

## 4. 今回の範囲外

- `veil-normalize.py` のロジック追加
- `veil-lint.py` の判定ロジック追加
- Web UI の機能変更
- `~/.veil/rules/` の実データ整理

## 5. 機能要件

### 5.1 判別順

語を登録対象へ送る前に、少なくとも次の順で判別できること。

1. 識別子か
2. 固有名か
3. 説明語か
4. プロジェクト固有語か
5. 境界が曖昧か

### 5.2 採用優先度

登録候補は、少なくとも次の順で優先できること。

1. 頻出で困り度が高い語
2. VEIL 運用そのものに効く基幹語
3. プロジェクト固有語
4. 低頻度語と境界が曖昧な語

### 5.3 保留ルール

次の語は即登録ではなく保留に回すことを明文化する。

- 固有名か一般語か迷う語
- 同じ概念に候補1が複数立つ語
- 既存ルールと別訳で衝突する語
- 頻度と困り度がまだ低い語

### 5.4 capture 接続

`veil-capture` 手順が、抽出した候補を次の流れで扱うこと。

1. 正規化
2. 判別
3. 優先度付け
4. 採用
5. 保留
6. 書き込み
7. 同期

## 6. 非機能要件

- 人向け説明は日本語優先で書く
- 既存の `~/.veil/rules/` 正本思想を崩さない
- README、設計書、skills、入口文書、UI マニュアルの記述が食い違わない

## 7. 完了条件

- 段階導入の原則が README にある
- 判別順と採用優先度が `docs/veil-design.md` にある
- `skills/codex/veil-capture/SKILL.md` と `skills/claude-code/veil-capture.md` に同じ判断順序がある
- `AGENTS.md` に高需要先行の運用原則がある
- `docs/manual.html` に自動推定と採用優先度の関係がある
- 保留条件が各面で一致している

## 8. リスク

- 文書だけで終わると実運用で抜ける可能性がある
- 採用優先度を厳しくしすぎると capture の成果が少なく見える
- 用語例を増やしすぎると再び混在を招く
