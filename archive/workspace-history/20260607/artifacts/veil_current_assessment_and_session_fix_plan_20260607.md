# VEIL 現状評価とこのセッションの修正提案

作成日: 2026-06-07

## 1. 未記録だった直近修正案の本線

現作業ツリー差分と直近コミットから見ると、直近で詰めていた修正案の本線は次である。

`VEIL の正本を ~/.veil/rules/ に固定し、veil-sync.py を app.py 非依存の同期器へ整理し、Web UI は補助機能へ位置付け直す。`

この解釈を支える根拠は次の通り。

- `README.md`
  - `~/.veil/rules/` を正本と明記している
  - Web UI をメインワークフローから外している
  - `veil-sync.py` は `~/.veil/rules/` と `behavior.md` のみを同期すると説明している
- `docs/veil-design.md`
  - 正本と補助を分けている
  - `behavior.md` 読み込みを前提化している
  - UI は補助と明記している
- `app.py`
  - 直近差分では `/vocab/prompt` と自動同期フックが削除対象になっている
- `veil-sync.py`
  - HTTP 経由での語彙取得と `--stdin` が削除対象になっている
  - `behavior.md` を同期に含める設計へ寄っている
- `skills/claude-code/veil-capture.md`
- `skills/codex/veil-capture/SKILL.md`
  - VEIL サーバー確認よりも `veil-sync.py` 実行確認を重視する方向へ寄っている

つまり、直近の修正案は「UI 起点の同期」から「rules 正本起点の同期」への責務の移し替えだったと見てよい。

## 2. 現在の VEIL に対する所感

### 良い点

- 発想の芯が明確
  - AI の出力後に都度直すのではなく、事前に語彙ルールを注入して出力を整える思想は筋が良い
- 依存が軽い
  - `app.py` と `veil-sync.py` が標準ライブラリ中心で、道具として維持しやすい
- スキル連携の方向が良い
  - `veil-capture` で候補抽出し、`~/.veil/rules/` を正本にする流れは再利用性が高い
- ドキュメント改修の方向も正しい
  - README と設計書はすでに「rules 正本・UI 補助」へ寄り始めている

### 気になる点

- authority のねじれがまだ残っている
  - ガバナンス文書では `shared/vocab.db`
  - 実装では repo 直下 `vocab.db`
  - 説明と実体が一致していない
- 旧設計の残骸が部分的に残っている
  - `docs/manual.html` に多言語時代のリンクと「現在の言語ペア」が残る
- 設計書と実装の粒度差がある
  - 設計書は「候補1だけが実変換に使われる」と読める
  - 実装は `p1` が空なら `p2` / `p3` にフォールバックする
- 役割分離は良くなっているが、まだ文書群が完全に同期していない
  - README
  - 設計書
  - マニュアル
  - ガバナンス文書
  が同じ前提を共有し切れていない

### 総評

現在の VEIL は、方向性はかなり良い。
ただし「設計の芯」は固まりつつある一方で、「正本の定義」「補助UIの位置付け」「DB の authority」がまだ完全に揃っていない。

壊れているプロジェクトではなく、設計整理の最終段に入っているプロジェクト、という見立てがいちばん近い。

## 3. このセッションで VEIL を良くする修正案

このセッションで効きが大きい順に並べる。

### 提案 1. authority を正式にそろえる

最優先。

決めるべきこと:

- `vocab.db` を repo 直下のまま正とするか
- `shared/vocab.db` に寄せるか

私の推奨:

- まずは repo 直下 `vocab.db` を正として文書をそろえる

理由:

- 実装がすでにそうなっている
- 今セッションでの修正量が小さい
- `vocab.db` は補助データという位置付けなので、大きな移設より authority 明確化の方が先

修正対象:

- `index/project-file-taxonomy.md`
- `index/project-template-adoption-packet.md`
- 必要なら `index/project-boundary-register.md`

### 提案 2. `docs/manual.html` の旧仕様を除去する

優先度高。

直すべき点:

- `/docs/manual/en.html` など存在しない多言語リンク
- 「現在の言語ペア」という旧仕様表現
- 多言語前提の説明が残っていないかの再点検

効果:

- 利用者向けの誤解が一気に減る
- 今の VEIL が何者かを最短で正しく伝えられる

### 提案 3. `docs/veil-design.md` と `ui/js/convert.js` の仕様差を解消する

優先度高。

論点:

- 実変換は `p1` のみか
- `p1` が空なら `p2` / `p3` フォールバックを認めるか

私の推奨:

- このセッションでは実装を変えず、まず設計書を現実に合わせて明記する

理由:

- いまの UI 操作感を壊さない
- まずは「現実の仕様」を正本化する方が安全
- 後で `p1` 厳格化をやるなら、別タスクとして判断できる

### 提案 4. `workspace/` の再構成内容を canonical 判断へ昇格する

優先度中。

やること:

- 直近修正案の本線を正式判断として README / 設計書 / index に反映する

反映する判断文の骨子:

- 正本は `~/.veil/rules/`
- `veil-sync.py` は rules と behavior を同期する
- `app.py` は補助UIであり同期のauthorityではない

### 提案 5. 「このセッションでやらないこと」も明記する

優先度中。

保留候補:

- `vocab.db` 自体を rules から再構成する仕組みの追加
- UI デザインの刷新
- `veil-capture` の抽出ロジック自動化強化
- 多数の新機能追加

理由:

- 今は新機能より整合性回復の方が投資対効果が高い

## 4. このセッションの推奨実行順

もっとも自然な順序は次。

1. `vocab.db` の authority を決める
2. `index/` の記述を現実へ合わせる
3. `docs/manual.html` の旧仕様を掃除する
4. `docs/veil-design.md` と `ui/js/convert.js` の仕様差を明文化する
5. 変更後に README / 設計書 / ガバナンスの整合確認をする

## 5. いま私が推す具体案

このセッションでそのまま進めるなら、次の修正が最も安全で効く。

### 採用案

- `vocab.db` は repo 直下の補助データとして文書を統一
- `~/.veil/rules/` を唯一の正本として固定
- `docs/manual.html` の旧多言語記述を除去
- `docs/veil-design.md` に `p1` / `p2` / `p3` フォールバック実装を明記

### この案の利点

- 既存実装をほぼ壊さない
- 説明と実装のズレを短時間で大きく減らせる
- 直近で詰めていた修正案の流れとも矛盾しない
- 次の大きな改善に入る前の土台固めになる

## 6. まとめ

いま明らかになったことは次の 3 点である。

1. 未記録だった直近修正案の本線は、rules 正本化と sync の独立化だった
2. 現在の VEIL は方向性はかなり良いが、authority と文書整合に未解決点が残っている
3. このセッションで最も価値が高いのは、新機能追加ではなく「正本・補助・同期責務」の整理である

次の着手としては、`index/` と `docs/manual.html` と `docs/veil-design.md` の整合修正が最適。
